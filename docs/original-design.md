# MindMap-MCTS 技能设计方案

> 一个给 Codex / Claude Code 等编码 Agent 通用的「思维导图牵引 × 轻量 MCTS」探索技能。
> 目标：让模型在面对复杂、多解、易走弯路的问题时，按一棵显式、可见、可牵引的思维树做系统化探索，而不是线性蛮干。

---

## 1. 问题与定位

线性 Agent 的典型失败模式是：选定一个方向就埋头走，即使真实信号已经在说「方向不对」，也因为沉没成本继续往下，最后绕一大圈。

本技能解决这一点，做法是把推理过程**外化成一个树文件**，并用轻量 MCTS（蒙特卡洛树搜索）来决定「下一步该深入哪个分支」。模型只负责「想」和「评」，搜索的调度交给一套确定性规则。

**何时触发**：任务复杂、解法不唯一、需要调试排查、有多个候选方案要权衡，或你发现模型在反复试错绕圈时。

**不该用的场景**：单步、确定性、一眼能看到答案的任务——上这套只会增加开销。

---

## 2. 核心理念

三句话概括：

1. **思维树是一个活文档**，不是藏在模型脑子里的隐式状态。模型每轮读它、改它、写回；你随时能看、能编辑它来牵引方向。
2. **轻量 MCTS 调度探索**：选择最值得深入的分支、发散少量子想法、评估、把价值回传——迭代逼近最优路径，而不是一次性铺开所有可能。
3. **用真实反馈代替随机模拟**：编码 Agent 能跑测试、读代码、grep、看日志，用真实信号给分支打分。这是它比纯推理强的地方，也是「轻量」与「不走弯路」的根本。

---

## 3. 架构总览

```
┌─────────────┐     调用算子      ┌──────────────┐
│    模型      │ ───────────────▶ │   树引擎      │
│ (想 / 评)    │                  │ (真相源+算子) │
└─────────────┘ ◀─────────────── └──────┬───────┘
      ▲          select 结果 / 树状态     │ 状态变更
      │                                  ▼
      │                          ┌──────────────┐
      └──── 牵引(编辑) ◀────────── │  可视化视图   │
            人                    │ (思维导图/TUI)│
                                  └──────────────┘
```

关键的职责划分——**这是整个设计的核心判断**：

| 环节 | 性质 | 谁来做 |
|------|------|--------|
| 选择（UCB 计算） | 确定性算法 | 代码 |
| 扩展（生成子想法） | 需要智能 | 模型 |
| 评估（打分 + 给证据） | 需要智能 | 模型（+ 真实探针） |
| 回传（更新 V/N） | 确定性算法 | 代码 |
| 可视化 | 渲染 | 代码 |

让模型手算 UCB 或手动回传统计，既慢又容易错。把机械部分交给代码，模型专注于不可替代的判断。

## 4. 数据结构：思维树

### 4.1 真相源

真相源用 **JSON 或 SQLite**，markdown 大纲和思维导图都只是它的渲染视图。
不要让模型用文本替换去改 markdown——那样统计会算错、格式会漂。模型改树只能通过树引擎的原子算子。

### 4.2 节点字段

| 字段 | 含义 |
|------|------|
| `id` | 唯一标识 |
| `parent` | 父节点 id |
| `content` | 这一节点代表的想法/假设/方案/待验证子问题 |
| `type` | 目标分解 / 假设 / 方案 / 验证动作 |
| `state` | 前沿 / 探索中 / 已证实 / 已剪枝 / 选定 |
| `V` | 价值估计，∈[0,1] |
| `N` | 访问次数 |
| `evidence` | 支撑当前 V 的简短证据（如「跑了 test_login，复现超时」） |

### 4.3 可读视图（markdown 大纲）

每节点一行，缩进表示父子。状态用符号：🌱前沿 / 🔍探索中 / ✓已证实 / ✗已剪枝 / ▶选定路径。

```
- [n1] (V=0.85 N=5 🔍) 修复登录偶发超时
  - [n2] (V=0.90 N=3 ✓) 假设：DB 连接池耗尽 — 日志见 "pool timeout"，已复现
    - [n5] (V=0.95 N=1 ▶) 方案：max 5→20 + 超时回收 — 压测下超时消失
  - [n3] (V=0.10 N=1 ✗) 假设：前端重试风暴 — grep 无重试逻辑，排除
  - [n4] (V=0.40 N=1 🌱) 假设：慢查询拖垮请求 — 待 EXPLAIN 验证
```

人和模型都能读，搜索往哪走、哪条路被压低，一眼可见。

## 5. 核心算法：轻量 MCTS 四步循环

每轮迭代做一次完整循环。

### 5.1 选择 Selection（代码）

从根往下，每层在子节点中选 score 最高者，直到到达一个前沿节点。

```
score = V + c · √( ln(N_parent) / N_child )
```

- `c`：探索系数，默认约 0.7（调大→更爱探索新分支；调小→更爱深挖已知好分支）
- `N_child = 0` 的节点视为 score = ∞（从未试过的方向必须先给一次机会）

口语版：优先深入目前最有希望的分支，但别忘了给还没认真试过的分支一次机会。

### 5.2 扩展 Expansion（模型）

对选中的前沿节点，生成 **2–3 个**子节点。要求它们**互斥、具体、可被验证**，而不是空泛的同义改写。宽度上限默认 3——一次铺开十几个是发散，不是搜索。

### 5.3 评估 Evaluation（模型 + 真实探针）—— 轻量化的灵魂

给每个新子节点打分 V∈[0,1]。**核心原则：优先用真实、廉价的探针拿信号，而不是凭空想象。**

- 编码任务就跑一个最小测试、读相关代码、grep 关键符号、看一处日志，用真实反馈判断这条路通不通。
- 真实证据远比「我觉得这个方案应该行」可靠，也正是它让搜索不再走弯路。
- 只有当真实探针成本太高或不适用时，才退回到模型自评估（基于已知信息估计成功可能性 / 信息增益 / 离目标距离）。
- 把支撑该分数的证据简短记到节点上，让价值可追溯、可被质疑。

这一步替代了标准 MCTS 里昂贵又不准的「随机 rollout」：我们不模拟到终局，而是用一两个高信号的真实探针来近似这条分支的前景。

**估值刻度参考**：

| V | 含义 |
|---|------|
| 0.9–1.0 | 真实证据已证实能到目标（测试通过、复现并定位） |
| 0.6–0.8 | 强信号支持，但仍需验证 |
| 0.3–0.5 | 不确定，证据中性 |
| 0.1–0.2 | 真实信号已基本证伪 |
| 0 | 死分支（确证不通），剪枝 |

### 5.4 回传 Backpropagation（代码）

把评估结果沿选择路径传回根：

- 每个祖先 `N += 1`
- 每个祖先 `V` 更新为其**子节点价值的最优值**（关心这条分支下最好的出路，不是平均；偏保守时可用均值）

于是一个分支下只要存在一条好路径，价值就被抬上来；若下面全是死路，价值塌下去，下一轮选择自然不再来。

---

## 6. 两个「不走弯路」的关键机制

**真实探针代替随机 rollout。** 标准 MCTS 的 rollout 在编码场景又贵又不准。换成「跑一次代码 / 读一段实现 / grep 一下」拿真实反馈，信号质量高得多——这正是编码 Agent 的独特优势。

**回退是自动的，不靠「下决心」。** 一旦某分支被真实探针证伪：V 置低、标 ✗、记下原因。价值回传后它自动失去竞争力，下一轮选择转向别处。被证伪的死分支**保留不删**，并附上「为什么不通」，防止之后绕回重试同一条死路。

---

## 7. 收敛与终止

满足任一条件即收敛：

- 存在一条从根到叶、价值 ≥ 0.85 且关键节点已被真实证据验证的路径 → 按它执行。
- 迭代预算耗尽（默认 5–8 轮）→ 选当前最优路径执行，并标注剩余不确定性。
- 多条路径价值接近且都未被证伪 → 这是信号：要么再设计一个能区分它们的廉价探针，要么把这个分叉暴露给人来决定（见 §9）。

收敛后，把选定路径写成清晰的执行计划，再开始真正动手实现。

## 8. 工具组件（按优先级）

### 8.1 树引擎（最该做）

真相源 + 原子算子。模型调函数，不手搓文本。

- 智能算子：`add_node(parent, content)`、`evaluate(id, V, evidence)`、`prune(id)`
- 确定性算子：`select_frontier()`（跑 UCB 挑节点）、`backprop(id)`（自动回传）
- 导出：`render_markdown()`、`render_json()`

### 8.2 实时可视化（三档，按省事程度）

- **零代码**：树写成 markmap 兼容 markdown，用 `markmap-cli` 的 watch 模式，或 VS Code 的 Markmap / Markdown Preview Enhanced 插件，存盘即刷新。最快上手。
- **轻量自建（推荐）**：文件监听（Python `watchdog` / Node `chokidar`）→ websocket 推送 → 浏览器用 `vis-network` 或 `Cytoscape.js` 渲染。把 MCTS 语义映射到视觉：**颜色=状态**（绿✓ / 红✗ / 灰🌱 / 蓝▶），**大小或边框=V 值**，一眼看出搜索往哪走、哪条路被压低。约 150 行。
- **终端 TUI**：用 `Rich` / `Textual` 画实时树，适合纯命令行流（Codex、Claude Code 本就在终端里），不用切窗口。

### 8.3 探针封装

把评估那步的「跑测试 / grep / 读日志」封成标准 `probe` 命令，结果结构化回写到节点证据里，保证价值可追溯。

### 8.4 预算护栏

一个计数器：迭代数、深度、token 超限即停，防搜索失控。

### 8.5 快照回放（可选）

每轮存一份树快照，事后可回放整个探索过程，复盘「为什么走了这条路」，便于调试 Agent 行为。

## 9. 人在环中：牵引这棵树

思维树是一个**你能看到也能改的文件**，这就是「牵引」的入口：

- 随时查看树，了解模型在想什么、走到哪、为什么放弃某条路。
- 想改方向就直接编辑：加一个你想让它考虑的节点、调高/调低某节点价值、把某分支标 ✗ 剪掉、在节点上写约束注释。模型下一轮读取并尊重你的编辑。
- 在关键分叉处，模型应主动停下把选项摆给你，而不是替你赌一个方向。

## 10. 部署

| 环境 | 做法 |
|------|------|
| **Claude Code** | 把流程写进 `CLAUDE.md`，或做成一个 skill 放进 `.claude/skills/` |
| **Codex** | 把核心循环写进项目根的 `AGENTS.md`，或在其中加指针引用本流程文档 |
| **通用** | 树文件是纯文本、与工具无关；任何能读指令文件的 Agent 都可引用本协议 |

---

## 11. 默认参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 分支宽度 | 2–3 | 一次发散的子节点数 |
| 最大深度 | 4–6 | 太深说明问题没拆好 |
| 迭代轮数 | 5–8 | 简单任务 2–3 轮足矣 |
| 探索系数 c | ≈0.7 | 控制探索 vs 利用 |

精神是「少量高信号探索」，不是穷举。按任务难度上下调整。

## 12. 实施路线

| 阶段 | 内容 |
|------|------|
| **MVP** | 树引擎（§8.1）+ 轻量自建可视化（§8.2 第二档）。这两件组合即可用。 |
| **增强** | 探针封装（§8.3）+ 预算护栏（§8.4） |
| **打磨** | 快照回放（§8.5）+ 调参与多场景验证 |

建议先在一个真实的复杂任务（如一次 bug 排查或一个架构选型）上跑 MVP，根据实际效果再调宽度、深度、迭代数和 c。
